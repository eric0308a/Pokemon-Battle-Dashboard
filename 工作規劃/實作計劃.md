# Pokemon Battle Dashboard 實作計劃

## 1. 計劃目標

本專案要做的是一個面向快節奏對戰情境的寶可夢輔助工具，核心價值是讓使用者在短時間內完成選怪、速度判定、傷害預估與對戰決策。這份文件把整體流程拆成「MVP -> 可用版本 -> 完整版本」三個層次，方便先做出可上手的產品，再逐步補齊進階能力。

目前已確認的產品方向：
- 以最新 Pokémon Champions 規則為主。
- Mega 進化是核心機制之一。
- 採純前端架構，部署到 GitHub Pages。
- 以 URL 參數與本地儲存做隊伍分享與跨裝置延續。
- 前端連線 PokeAPI 時，優先採用 `pokeapi-js-wrapper` 來統一讀取與快取流程。

## 2. 產品原則

### 2.1 優先順序

1. 先能用，再變強。
2. 先做核心決策，再做完整編輯功能。
3. 先鎖定單一規則集，再考慮多格式擴充。
4. 先把純計算層做穩，再把 UI 做漂亮。

### 2.2 設計原則

- 對戰中每一步操作都要短。
- 資料要離線可用，不依賴即時外部 API。
- 核心計算要可測試、可重用、可替換。
- UI 以手機優先，因為實際使用情境多半是快速查詢。
- PokeAPI 存取要集中在單一 wrapper 層，避免元件直接散發 API 呼叫。

## 3. MVP 定義

MVP 的目標不是做完整編輯器，而是讓使用者在實戰中能立刻得到有價值的判斷。

### 3.1 MVP 必備功能

- 建立或載入一組我方隊伍。
- 選擇對手寶可夢與其主要資訊。
- 顯示速度比較與先後手判定。
- 顯示屬性相剋結果。
- 顯示傷害區間與確一、確二等判定。
- 支援 Mega 進化前後切換。
- 支援隊伍 URL 分享與還原。
- 支援 LocalStorage 保存常用隊伍。

### 3.2 MVP 不先做的項目

- 完整戰報紀錄。
- 多格式同時支援。
- AI 建議與自動配招。
- 社群排行榜或分享牆。
- 複雜的歷史回放系統。
- 進階 PWA 離線資源管理。

### 3.3 MVP 成功標準

- 使用者能在 1 分鐘內完成一次基本對戰判斷。
- 重新整理頁面後，狀態可從 URL 或本機儲存恢復。
- 主要計算結果在單元測試中穩定通過。
- 手機螢幕上可正常操作，不需要精細拖拉或小按鈕點選。

## 4. 分階段實作流程

## Phase 0. 規格鎖定與資料建模

### 目標

先把規則邊界與資料欄位定義清楚，避免後面一直重構。

### 工作項目

- 確認支援的對戰格式與規則版本。
- 定義 Mega 進化的資料結構與切換方式。
- 定義寶可夢、招式、道具、特性、努力值、個體值、性格等欄位。
- 定義傷害計算與速度判定所需的最小資料集合。
- 設計 URL 編碼與版本號格式。
- 決定本地快照資料的來源與更新策略。

### 交付物

- 資料型別定義文件。
- MVP 功能範圍說明。
- 核心資料欄位清單。
- URL 格式草案。

### 驗收條件

- 能清楚回答「MVP 支援什麼、不支援什麼」。
- 核心資料欄位可對應到 UI 與計算層。
- 未來版本能靠版本號辨識資料相容性。

## Phase 1. 專案骨架與資料管線

### 目標

建立可持續開發的基礎架構，並先把靜態資料準備好。

### 工作項目

- 建立 Vue 3 專案骨架。
- 加入 TypeScript、Vite、Pinia、Tailwind CSS。
- 規劃元件、狀態、工具函式與資料目錄。
- 安裝並整合 `pokeapi-js-wrapper`，作為前端讀取 PokeAPI 的統一入口。
- 建立 PokeAPI 存取封裝層，集中管理 `Pokedex` 實例、快取設定與錯誤處理。
- 撰寫從 PokeAPI 或等效來源抓取資料的腳本。
- 將資料整理成本地 JSON 快照。
- 建立基本測試框架。

### 交付物

- 專案可啟動的骨架。
- `pokeapi-js-wrapper` 的整合與初始化封裝。
- 可重複產生資料快照的腳本。
- 本地靜態資料檔案。
- 基本測試環境。

### 驗收條件

- 資料可離線載入。
- 主要依賴可安裝、可啟動、可建置。
- 基礎測試可以執行。
- PokeAPI 讀取流程不會散落在多個元件中，且能透過同一層封裝控制快取與錯誤。

## Phase 2. 核心計算引擎

### 目標

把速度、屬性、傷害與 Mega 進化等規則做成可獨立測試的純邏輯層。

### 工作項目

- 實作屬性相剋查表與倍率計算。
- 實作速度判定邏輯。
- 實作 Mega 進化前後能力值切換。
- 整合傷害計算公式。
- 產出傷害範圍、命中與擊殺機率描述。
- 加入資料驗證與錯誤處理。

### 交付物

- `damage`、`speed`、`type effectiveness` 等純函式模組。
- 計算結果資料結構。
- 核心公式測試。

### 驗收條件

- 固定輸入可得到穩定輸出。
- Mega 切換後的結果與規則一致。
- 任何資料錯誤都能被明確攔截並回報。

## Phase 3. MVP 介面

### 目標

先做最短操作路徑，讓使用者能快速完成一場對戰判斷。

### 工作項目

- 建立主畫面與戰鬥畫面。
- 建立我方隊伍選擇器。
- 建立對手選擇器。
- 建立速度比較區塊。
- 建立屬性相剋顯示區塊。
- 建立傷害結果顯示區塊。
- 建立 Mega 進化切換操作。
- 優化手機版排版與可點擊區域。

### 交付物

- 可直接操作的 MVP 畫面。
- 基本互動流程完成。
- 行動版可用介面。

### 驗收條件

- 使用者能快速切換寶可夢並看到結果更新。
- 主要資訊不需要翻頁就可看見。
- 小螢幕下仍能順暢操作。

## Phase 4. 狀態管理與分享

### 目標

讓使用者能保存、分享與恢復隊伍設定。

### 工作項目

- 使用 Pinia 管理對戰狀態。
- 使用 LocalStorage 保存常用隊伍。
- 實作 URL 編碼與解碼。
- 加入資料版本檢查。
- 加入解碼失敗與格式不相容提示。

### 交付物

- 隊伍分享連結。
- 本機常用隊伍清單。
- 恢復狀態流程。

### 驗收條件

- 重新整理後可恢復狀態。
- 複製 URL 到另一台裝置可還原隊伍。
- 不合法資料會給出可理解的錯誤。

## Phase 5. 測試與穩定化

### 目標

確認 MVP 的核心功能在真實使用情境下穩定可用。

### 工作項目

- 為計算引擎撰寫單元測試。
- 為 URL 編解碼撰寫單元測試。
- 為資料轉換撰寫單元測試。
- 建立最少一條端到端流程測試。
- 修正手機版與桌面版互動問題。

### 交付物

- 單元測試。
- E2E 測試。
- 穩定版本的 MVP。

### 驗收條件

- 核心測試長期可重複通過。
- 主要流程無阻斷錯誤。
- 行動裝置與桌面裝置皆可操作。

## 8. 風險與注意事項

### 8.1 規則版本風險

Pokémon Champions 的規則若與既有主系列差異很大，計算公式與資料欄位可能需要重新調整。因此一開始就應該把規則邏輯集中在單一層，不要散落在 UI 裡。

### 8.2 URL 長度風險

如果將太多隊伍或戰鬥狀態塞進 URL，長度可能超出瀏覽器限制。MVP 建議只編碼必要隊伍資料，複雜資訊留在本機儲存或之後再擴充。

### 8.3 資料更新風險

PokeAPI 或其他資料來源可能更新不一致，因此最好使用版本化的本地快照，而不是在對戰當下即時查詢外部服務。

### 8.4 Wrapper 相容風險

`pokeapi-js-wrapper` 是 browser-first 的 wrapper，適合直接放在前端專案中，但需要注意它的快取與 service worker 行為。若要啟用圖片快取，部署時必須把 service worker 以網站根目錄方式提供，這和 GitHub Pages 的靜態部署模式相容，但仍需在實作時驗證路徑與 scope。

### 8.5 使用情境風險

實際使用多半是快速判斷，操作成本不能太高。如果畫面層級太深、步驟太多，會直接降低工具價值。

## 9. 下一步建議

1. 先把 MVP 的資料型別與規則邊界定稿。
2. 再建立專案骨架、`pokeapi-js-wrapper` 整合與靜態資料管線。
3. 然後先做核心計算引擎，最後才是 UI。
